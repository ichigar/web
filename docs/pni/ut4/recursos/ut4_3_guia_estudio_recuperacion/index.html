    Guía de estudio — Recuperación UT4-AE1: VLAN + SVI + Enrutamiento¶

    Cómo usar esta guía¶

    Esta guía está pensada para los alumnos que necesitan preparar el examen de recuperación de la prueba práctica UT4-AE1. Cubre todos los apartados de la prueba con explicaciones, comandos y los errores más frecuentes detectados en las correcciones.

    Contenido de cada sección:

    • Sección 1 — Diagrama de direccionamiento: cómo calcular las IPs de todas las interfaces, qué se suele olvidar (subinterfaces del router) y cómo evitar el error más frecuente con prefijos /16.
    • Sección 2 — Conceptos previos: teoría esencial sobre switch L2 vs L3, VLANs, trunks, SVIs, router-on-a-stick y enrutamiento estático. Incluye el catálogo completo de comandos show que pueden pedirse.
    • Sección 3 — Resolución guiada: recorrido paso a paso por cada apartado (0a–8f) con explicación, ejemplo de comandos, errores frecuentes y preguntas de reflexión con respuesta oculta.
    • Sección 4 — Tabla de errores frecuentes: resumen de los errores más comunes detectados en las correcciones, con su causa y cómo evitarlos.
    • Sección 5 — Checklist de autoevaluación: lista de verificación para repasar antes de entregar, organizada por bloques temáticos.
    • Sección 6 — Anexo de configuración: todos los comandos a ejecutar en cada dispositivo (PCs, SW_1, SW_2, SW_3, SWL3_1, R_1) con los valores del examen original como referencia.

    Método de estudio recomendado:

    1. Lee la explicación conceptual de cada apartado (Sección 2).
    2. Consulta el ejemplo de comandos en la resolución guiada (Sección 3).
    3. Reproduce la configuración completa en Packet Tracer desde cero usando el anexo (Sección 6).
    4. Verifica cada paso con el comando show correspondiente.
    5. Responde mentalmente las preguntas de reflexión antes de pasar al siguiente paso.

    La recuperación será similar, pero distinta

    Los números de VLAN, las redes IP, los nombres de interfaz y los comandos de verificación pedidos pueden cambiar respecto a la prueba original. Lo que no cambia es la lógica de configuración. Esta guía te enseña el método, no los valores de memoria.

    1. El diagrama de direccionamiento: cómo calcularlo y qué no olvidar¶

    Antes de configurar ningún dispositivo en Packet Tracer debes tener claras las IPs de todas las interfaces que llevan dirección IP. Es lo que pide el paso 0b y es la base de todo lo que viene después.

    1.1 Cómo leer los criterios de asignación¶

    Lee siempre el enunciado antes de calcular. Los criterios habituales son:

    Tipo de interfaz Criterio de asignación Ejemplo
    PCs y dispositivos finales IP de la red + número del equipo como host PCC_2 → 10.100.0.2
    SVI del switch L3 (gateway de VLAN) Última IP disponible del prefijo Vlan30 → 192.168.30.254
    Subinterfaces del router (RoaS) Última IP disponible del prefijo Gig0/1.100 → 10.100.255.254
    Enlace punto a punto (/30) De izquierda a derecha según el diagrama SWL3_1 → 10.0.0.1 · R_1 → 10.0.0.2

    1.2 Interfaces que se suelen olvidar¶

    Olvido más frecuente: las subinterfaces del router-on-a-stick

    Las subinterfaces del router (Gig0/1.100, Gig0/1.200…) deben aparecer en el diagrama con sus IPs. Es el error más habitual en el paso 0b: se dibuja la IP del enlace /30 y las SVIs del L3, pero se olvidan las subinterfaces del router.

    Antes de entregar el diagrama comprueba que aparecen todas estas IPs:

    • SWL3_1 Gig0/2 y R_1 Gig0/0 (enlace /30)
    • R_1 Gig0/1.100 y Gig0/1.200 (subinterfaces RoaS) ← las que más se olvidan
    • SWL3_1 Vlan30 y Vlan40 (SVIs del switch L3)

    1.3 Cómo calcular la última IP disponible¶

    Error más frecuente: confundir la última IP del prefijo

    Para una red /16 como 10.100.0.0/16:

    • Broadcast = 10.100.255.255
    • Última host = 10.100.255.254
    • ¡NO es 10.100.0.254! — esa sería la última del primer /24 dentro del /16.
    Prefijo Broadcast Última host
    10.100.0.0/16 10.100.255.255 10.100.255.254
    192.168.30.0/24 192.168.30.255 192.168.30.254
    10.0.0.0/30 10.0.0.3 10.0.0.2 (host 2 de 2)

    1.4 Cómo generar la captura del diagrama¶

    1. Abre la vista Logical en Packet Tracer.
    2. Etiqueta cada interfaz con su IP directamente sobre el canvas (notas de texto o etiquetas del dispositivo).
    3. Haz la captura cuando todas las interfaces con IP estén visibles y etiquetadas.

    Verificación rápida

    Ejecuta show ip interface brief en SWL3_1 y en R_1. Las IPs deben coincidir exactamente con las del diagrama. Si hay discrepancia, corrígela antes de continuar con el resto de pasos.

    2. Conceptos previos imprescindibles¶

    2.1 Switch de capa 2 vs switch de capa 3¶

    Antes de empezar a configurar, es fundamental saber qué dispositivo tienes delante.

    Característica Switch L2 (ej. SW_2, SW_3) Switch L3 (ej. SWL3_1)
    Función principal Conmutar tramas (capa 2) Conmutar tramas + enrutar paquetes
    VLANs Sí Sí
    Puertos trunk/access Sí Sí
    SVIs con IP Solo para gestión (VLAN 1) Para enrutamiento inter-VLAN
    ip routing No disponible Obligatorio para enrutar
    Puertos routed (no switchport) No Sí
    Tabla de enrutamiento No Sí (show ip route)

    Concepto clave

    Un switch L3 no enruta automáticamente por el hecho de tener SVIs configuradas. Necesita el comando ip routing para activar el plano de enrutamiento. Sin él, las SVIs solo sirven para gestión.

    Reflexión

    En tu topología, ¿qué dispositivos son L2 y cuál es L3? ¿Por qué no tiene sentido ejecutar ip routing en un switch L2?

    Respuesta

    SW_2 y SW_3 son L2; SWL3_1 es L3. Un switch L2 trabaja exclusivamente con direcciones MAC y no tiene tabla de enrutamiento IP. El comando ip routing activa el motor de enrutamiento de capa 3, que un L2 no posee; ejecutarlo en un L2 daría error o no tendría ningún efecto.

    2.2 VLANs: en qué switch crearlas y qué tipo de puerto usar¶

    Tipos de puerto:

    • Access: conecta un único dispositivo final (PC, impresora). El tráfico no lleva etiqueta VLAN.
    • Trunk: conecta dos switches entre sí, o un switch con un router. Transporta tráfico de múltiples VLANs con etiqueta 802.1Q.

    Dónde crear cada VLAN:

    Solo se crea una VLAN en un switch si ese switch tiene dispositivos de esa VLAN conectados directamente, o si necesita transportar esa VLAN por un trunk.

    Error frecuente

    Crear VLANs de redes remotas en un switch que no tiene dispositivos de esas VLANs. Por ejemplo, crear VLAN 30 y VLAN 40 en SW_3 cuando los PCs de esas redes están conectados a SWL3_1, no a SW_3. Solo las VLANs que SW_3 necesita transportar (100 y 200) deben existir en SW_3.

    Reflexión

    Mira la topología. Traza el camino que sigue un paquete de PCC_2 a PCA_1. ¿Qué switches atraviesa? ¿Qué VLANs necesita conocer cada uno?

    Respuesta

    PCC_2 → SW_3 (VLAN 100) → SW_2 (trunk VLAN 100) → R_1 (subinterfaz Gig0/1.100, enruta) → Gig0/0 → SWL3_1 (enruta por Vlan30) → SW_1 → PCA_1.

    • SW_3: necesita VLAN 100 para transportarla al trunk hacia SW_2.
    • SW_2: necesita VLAN 100 (access de PCC) y trunk activo hacia R_1.
    • SWL3_1: necesita VLAN 30 (SVI gateway de PCA_1).
    • SW_1: necesita VLAN 30 con el puerto access de PCA_1.

    2.3 Puertos trunk: cuándo y dónde configurarlos¶

    Un puerto debe estar en modo trunk cuando transporta tráfico de más de una VLAN. Eso ocurre en:

    • Los uplinks entre switches (el cable que conecta un switch con otro switch).
    • La interfaz del switch que conecta con el router-on-a-stick (el router que hace subinterfaces para enrutar entre VLANs).

    Error más frecuente

    No configurar trunk en la interfaz del switch que conecta al router. Sin ese trunk, el router recibe el tráfico sin etiquetar y no puede distinguir a qué VLAN pertenece cada paquete. El resultado es que el enrutamiento inter-VLAN falla por completo.

    Error también frecuente

    Configurar trunk en los puertos de acceso (los que van a los PCs, por ejemplo Fa0/1 o Fa0/10) en lugar de en los uplinks (Gig0/1, Gig0/2). Consecuencia: esos puertos dejan de ser access y las VLANs pierden sus puertos asignados en el show vlan brief.

    Comandos para configurar y verificar:

    ! Configurar trunk
interface GigabitEthernet0/1
 switchport mode trunk

! Verificar trunks activos
show interfaces trunk

! Verificar estado de VLANs y puertos
show vlan brief

    Comandos de verificación alternativos

    En la recuperación puede pedirse show interfaces trunk en lugar de show vlan brief. Ambos son válidos para verificar la configuración de trunks. show interfaces trunk muestra directamente qué interfaces están en modo trunk y qué VLANs transportan.

    2.4 SVI + ip routing en el switch L3¶

    Una SVI (Switch Virtual Interface) es una interfaz lógica asociada a una VLAN. En un switch L3, se usa para asignar una IP a cada VLAN y actuar como puerta de enlace de los dispositivos de esa VLAN.

    ip routing                          ! Activa el enrutamiento
interface vlan 30
 ip address 192.168.30.254 255.255.255.0
 no shutdown
interface vlan 40
 ip address 192.168.40.254 255.255.255.0
 no shutdown

    Error frecuente

    Olvidar el comando ip routing. Las SVIs aparecerán configuradas y el show ip interface brief mostrará las IPs, pero el switch no enrutará entre VLANs. Los pings inter-VLAN fallarán.

    Verificación

    Ejecuta show ip route. Si ves las redes directamente conectadas (C) de las VLANs, ip routing está activo. Si el comando devuelve error o tabla vacía, falta ip routing.

    Reflexión

    ¿Por qué el estado de una SVI puede aparecer como up/down aunque la hayas configurado correctamente? ¿Qué significa que el protocolo esté down?

    Respuesta

    Status up / Protocol down: la interfaz lógica existe y no está en shutdown (capa 1 activa), pero no hay ningún puerto físico activo asignado a esa VLAN en el switch (protocolo de capa 2 caído). Para que suba a up/up debe existir al menos un puerto access de esa VLAN con un dispositivo conectado y activo. En Packet Tracer: asegúrate de que el PC está conectado al puerto correcto y que la interfaz del PC está up.

    2.5 Router-on-a-stick¶

    El router-on-a-stick permite enrutar entre VLANs usando subinterfaces en una única interfaz física. Cada subinterfaz se asocia a una VLAN mediante encapsulation dot1Q.

    interface GigabitEthernet0/1.100
 encapsulation dot1Q 100
 ip address 10.100.255.254 255.255.0.0
interface GigabitEthernet0/1.200
 encapsulation dot1Q 200
 ip address 10.200.255.254 255.255.0.0

    Error frecuente

    Omitir encapsulation dot1Q. Sin este comando, el router no sabe a qué VLAN pertenece el tráfico que llega por esa subinterfaz. Las subinterfaces quedarán configuradas pero no funcionarán.

    Concepto clave

    La interfaz física (Gig0/1) no necesita IP. Son las subinterfaces las que tienen IP. La interfaz física solo necesita estar en no shutdown.

    Consejo

    Verifica con show ip interface brief que las subinterfaces aparecen up/up. Si aparecen administratively down, la interfaz física (Gig0/1) probablemente tiene shutdown.

    2.6 Enrutamiento estático¶

    El comando ip route agrega manualmente una ruta a la tabla de enrutamiento.

    ip route <red_destino> <máscara> <next-hop>
    • red_destino: la red a la que quiero llegar
    • máscara: la máscara de esa red
    • next-hop: la IP del siguiente salto (el dispositivo al que envío el paquete)

    Error frecuente

    Teclear los comandos ip route pero que las rutas no aparezcan en el show ip route. Esto ocurre cuando el next-hop no es alcanzable (falta conectividad al enlace punto a punto) o cuando hay un error tipográfico en la IP. El comando puede aceptarse sin error pero la ruta queda inactiva.

    Verificación

    Ejecuta show ip route. Las rutas estáticas activas aparecen con la letra S. Si no ves la S junto a la red destino, la ruta no está activa aunque hayas tecleado el comando.

    2.7 Regla de la última IP disponible¶

    Las puertas de enlace y las IPs de subinterfaces/SVIs se asignan habitualmente a la última IP host disponible de la red.

    Prefijo Última IP host Broadcast
    /24 x.x.x.254 x.x.x.255
    /16 x.x.255.254 x.x.255.255
    /30 Depende del bloque Depende del bloque

    Cómo calcularlo para cualquier red:

    El broadcast de una red es la dirección con todos los bits de host en 1. La última IP host es el broadcast menos 1.

    • 10.100.0.0/16 → broadcast = 10.100.255.255 → última host = 10.100.255.254 ✓
    • 10.100.0.0/16 → NO es 10.100.0.254 (esa sería la última del primer bloque /24, no del /16)

    Error frecuente

    Asignar 10.100.0.254 en lugar de 10.100.255.254 para una red /16. Es el error más común: confundir la última IP del /16 con la última del primer /24 dentro de ese /16.

    Reflexión

    ¿Cuál sería la última IP host de la red 172.16.0.0/12? ¿Y de 10.0.0.0/30?

    Respuesta
    • 172.16.0.0/12: broadcast = 172.31.255.255 → última host = 172.31.255.254
    • 10.0.0.0/30: rango = 10.0.0.0–10.0.0.3, broadcast = 10.0.0.3, hosts válidos = 10.0.0.1 y 10.0.0.2 → última host = 10.0.0.2

    2.8 Catálogo de comandos show relevantes¶

    Estos son los comandos de verificación que pueden aparecer en la prueba. Cualquiera de ellos puede pedirse en cualquier apartado.

    Comando Qué muestra Cuándo pedirlo
    show vlan brief VLANs existentes, estado y puertos asignados Tras crear VLANs y asignar puertos access
    show interfaces trunk Interfaces en modo trunk, VLANs permitidas/activas Para verificar que los trunks están configurados correctamente
    show interfaces <int> switchport Modo del puerto (access/trunk), VLAN asignada, si switchport está habilitado Para verificar estado de una interfaz específica; en puertos routed muestra "Switchport: Disabled"
    show ip interface brief Estado up/down de todas las interfaces con sus IPs Tras configurar SVIs, subinterfaces o IPs en interfaces
    show ip route Tabla de enrutamiento: rutas directas (C), estáticas (S), etc. Para confirmar que las rutas estáticas están activas
    show running-config Configuración activa completa del dispositivo Para verificar cualquier configuración cuando no hay un show específico

    Consejo

    Memoriza qué muestra cada comando. Si en la recuperación te piden un show distinto al de la prueba original, no entres en pánico: busca cuál de estos es el más adecuado para lo que se quiere verificar.

    3. Resolución guiada paso a paso¶

    A continuación se recorre cada apartado de la prueba. Los bloques de código muestran los valores del examen original como ejemplo; en la recuperación pueden cambiar.

    Paso 0a — Tabla de redes: máscaras¶

    Qué se pide: completar una tabla con las máscaras de subred de las redes de la topología.

    Concepto

    La máscara indica cuántos bits son de red. Debes conocer la equivalencia entre la notación CIDR (/16, /24, /30) y la decimal punteada.

    CIDR Decimal
    /30 255.255.255.252
    /24 255.255.255.0
    /16 255.255.0.0
    /12 255.240.0.0
    /8 255.0.0.0

    Ejemplo (prueba original):

    Red CIDR Máscara
    192.168.30.0 /24 255.255.255.0
    192.168.40.0 /24 255.255.255.0
    10.100.0.0 /16 255.255.0.0
    10.200.0.0 /16 255.255.0.0
    10.0.0.0 /30 255.255.255.252

    Error frecuente

    Dejar alguna máscara en blanco o poner 255.255.255.0 para un /16. Todas las filas deben estar completas y correctas.

    Reflexión

    ¿Por qué el enlace punto a punto entre dos routers/switches suele ser /30? ¿Cuántos hosts admite una /30?

    Respuesta

    Una /30 admite exactamente 2 hosts (2² − 2 = 2). Para un enlace P2P solo se necesitan dos IPs (una por extremo), y usar /30 evita desperdiciar un bloque más grande. Es la máscara más eficiente para este tipo de enlace.

    Paso 0b — Diagrama de direccionamiento¶

    Qué se pide: completar el diagrama de topología con las IPs de todas las interfaces que tienen dirección IP asignada.

    Concepto

    En una topología mixta con VLANs y enrutamiento, las interfaces con IP son: - IPs de PCs: red + número de equipo como host (ej. PCC_2 → 10.100.0.2) - SVIs del switch L3: última IP disponible del prefijo de cada VLAN. Poner en la captura junto al mismo. - Subinterfaces del router (RoaS): última IP disponible del prefijo de cada VLAN - Enlace punto a punto: IPs consecutivas del bloque /30

    Ejemplo (prueba original):

    SWL3_1 Gig0/2  → 10.0.0.1/30
R_1 Gig0/0     → 10.0.0.2/30
R_1 Gig0/1.100 → 10.100.255.254/16   ← última del /16
R_1 Gig0/1.200 → 10.200.255.254/16   ← última del /16
SWL3_1 Vlan30  → 192.168.30.254/24   ← última del /24
SWL3_1 Vlan40  → 192.168.40.254/24   ← última del /24
PCC_2          → 10.100.0.2/16
PCD_2          → 10.200.0.2/16
PCA_2          → 192.168.30.2/24
PRB_2          → 192.168.40.2/24

    Error frecuente

    Asignar 10.100.0.254 en lugar de 10.100.255.254 para una subinterfaz /16. Recuerda: para un /16, la última IP host está en el octeto 3 con valor 255, no 0.

    Reflexión

    ¿Por qué las subinterfaces del router y las SVIs del L3 usan la última IP disponible y no la primera?

    Respuesta

    Es una convención de diseño que facilita identificar visualmente el gateway: la primera IP (.1 o .0.1) queda libre para el primer host, y la última (.254 o .255.254) se reserva para el router/gateway. No hay razón técnica obligatoria, pero el enunciado lo especifica explícitamente y es práctica habitual en entornos Cisco.

    Paso 1 — Configuración IP en un PC¶

    Qué se pide: captura de un PC con IP, máscara y puerta de enlace correctas.

    Concepto

    La puerta de enlace de un PC debe ser la IP de la SVI o subinterfaz que actúa como gateway de esa VLAN. Si el gateway está mal, el PC no podrá comunicarse con otras redes aunque su IP sea correcta.

    Configura todos los PCs

    El examen pide captura de un solo PC, pero todos los PCs deben estar configurados para que los pings de pasos posteriores funcionen. Configura cada uno en Desktop → IP Configuration:

    PC IP Máscara Gateway
    PCA_1 192.168.30.1 255.255.255.0 192.168.30.254
    PCA_2 192.168.30.2 255.255.255.0 192.168.30.254
    PCB_1 192.168.40.1 255.255.255.0 192.168.40.254
    PRB_2 192.168.40.2 255.255.255.0 192.168.40.254
    PCC_1 10.100.0.1 255.255.0.0 10.100.255.254
    PCC_2 10.100.0.2 255.255.0.0 10.100.255.254
    PCD_1 10.200.0.1 255.255.0.0 10.200.255.254
    PCD_2 10.200.0.2 255.255.0.0 10.200.255.254

    Ejemplo (prueba original, PCC_2 en VLAN 100 /16):

    IPv4 Address  : 10.100.0.2
Subnet Mask   : 255.255.0.0
Default Gateway: 10.100.255.254

    Error frecuente

    Poner como gateway IP que no es la última de la red (10.100.0.254 en lugar de 10.100.255.254 ya que la máscara es /16) o una IP de otra red.

    Verificación

    Desde el PC, haz ping a la puerta de enlace. Si responde, la configuración IP es correcta.

    Reflexión

    ¿Qué ocurre si la IP del PC es correcta pero el gateway es incorrecto? ¿Puede el PC comunicarse con otro PC de la misma VLAN? ¿Y con uno de otra VLAN?

    Respuesta
    • Misma VLAN: sí puede comunicarse. Los PCs del mismo segmento se alcanzan directamente en capa 2, sin necesidad de gateway.
    • Otra VLAN: no puede. Para enviar a otra red el PC necesita reenviar el paquete al gateway; si es incorrecto, la trama nunca llega al router/switch L3 y la comunicación falla.

    Paso 2a — Crear VLANs en el switch de acceso¶

    Qué se pide: captura mostrando la creación de VLANs en un switch L2.

    Concepto

    Solo se crean las VLANs que ese switch necesita. Un switch necesita una VLAN si: - Tiene dispositivos de esa VLAN conectados directamente, o - Necesita transportarla por un trunk hacia otro switch.

    No se crean VLANs de redes remotas que no pasan por ese switch.

    Configuración en todos los switches

    El examen pide captura en un switch, pero las VLANs y los puertos access deben configurarse en todos los switches que las necesitan:

    • SW_2: VLANs 100 y 200 → Fa0/1 (VLAN 100), Fa0/10 (VLAN 200)
    • SW_3: VLANs 100 y 200 → Fa0/1 (VLAN 100), Fa0/10 (VLAN 200)
    • SW_1: VLANs 30 y 40 → Fa0/1 y Fa0/2 (VLAN 30), Fa0/10 y Fa0/11 (VLAN 40)
    • SWL3_1: VLANs 30 y 40 sin puertos access (los PCs de esas VLANs están en SW_1; las VLANs se crean aquí para que las SVIs del paso 4a puedan levantarse)

    El ejemplo de comandos que sigue corresponde al switch de acceso principal. Repite el proceso con los valores correspondientes en el resto.

    ! En el switch L2 de acceso
vlan 100
 name VLAN_100
vlan 200
 name VLAN_200
exit

    Error frecuente

    Crear VLANs innecesarias (por ejemplo, VLAN 30 y VLAN 40 en SW_3 cuando esas redes pertenecen al bloque de SWL3_1). Aunque no penaliza directamente, demuestra confusión sobre qué dispositivos hay en cada segmento.

    También es error usar la GUI de Packet Tracer (pestaña Config) en lugar de la CLI. La GUI no crea las VLANs correctamente en todos los escenarios.

    Comandos de verificación

    show vlan brief
show running-config

    Reflexión

    Si creas VLAN 100 en un switch pero no asignas ningún puerto en modo access a esa VLAN, ¿qué verás en el show vlan brief? ¿Tendrá algún efecto en la conectividad?

    Respuesta

    El show vlan brief mostrará VLAN 100 como active pero con la columna Ports vacía. No tendrá efecto en la conectividad: los PCs conectados a puertos que sigan en VLAN 1 (default) no podrán comunicarse con otros equipos de VLAN 100 aunque la VLAN exista.

    Paso 2b — Verificar VLANs con show vlan brief¶

    Qué se pide: captura del show vlan brief mostrando las VLANs con sus puertos asignados.

    Concepto

    El show vlan brief muestra el estado de cada VLAN y los puertos access asignados. Un puerto en modo trunk no aparece en esta lista (los trunks transportan todas las VLANs pero no se muestran como asignados a ninguna en particular).

    Salida esperada (ejemplo):

    VLAN Name                             Status    Ports
---- -------------------------------- --------- -------------------------------
1    default                          active    ...
100  VLAN_100                         active    Fa0/1
200  VLAN_200                         active    Fa0/10

    Comandos de verificación alternativos

    En la recuperación puede pedirse show interfaces trunk para verificar los trunks, o show vlan brief para los puertos access. Prepara ambos.

    Reflexión

    Si Fa0/1 aparece en la columna de VLAN 1 en lugar de VLAN 100, ¿qué falta configurar?

    Respuesta

    Falta asignar el puerto en modo access a VLAN 100: 

    interface FastEthernet0/1
 switchport mode access
 switchport access vlan 100

    Paso 3a — Configurar puertos trunk en SW_2¶

    Qué se pide: captura de la configuración de los puertos trunk en el switch que conecta al router y a otros switches.

    Concepto

    Los trunks van en los uplinks: las interfaces que conectan con el router (RoaS) y con otros switches. En ningún caso van en puertos que conectan a PCs.

    Trunks en todos los switches

    El examen muestra SW_2, pero los trunks deben configurarse en todos los uplinks de la topología. Además de los dos trunks de SW_2, necesitas:

    • SW_3 Gig0/1 → trunk hacia SW_2
    • SW_1 Gig0/1 → trunk hacia SWL3_1
    • SWL3_1 Gig0/1 → trunk hacia SW_1

    En cada caso el comando es el mismo: switchport mode trunk en la interfaz de uplink.

    ! En SW_2
interface GigabitEthernet0/1
 switchport mode trunk        ! uplink hacia R_1 (router-on-a-stick)

interface GigabitEthernet0/2
 switchport mode trunk        ! uplink hacia SW_3

    Error más frecuente

    No configurar trunk en Gig0/1 (el uplink hacia el router). Sin este trunk, el router recibe tráfico sin etiquetar y no puede enrutar entre VLANs. Es el error que más impacta en la funcionalidad de toda la topología.

    Error también frecuente

    Configurar Fa0/1 y Fa0/10 como trunk. Estos son puertos de acceso hacia PCs. Al ponerlos en modo trunk, dejan de pertenecer a sus VLANs y el show vlan brief deja de mostrar los puertos asignados.

    Comandos de verificación

    show interfaces trunk
show vlan brief
    Con show interfaces trunk verás exactamente qué interfaces están en modo trunk y qué VLANs están activas en ellas.

    Reflexión

    Si Gig0/1 no está en modo trunk, ¿qué le pasa al tráfico de VLAN 100 cuando llega al router? ¿Lo puede procesar la subinterfaz Gig0/1.100?

    Respuesta

    Si Gig0/1 está en modo access (o sin configurar), el switch solo envía al router el tráfico sin etiquetar de la VLAN nativa (normalmente VLAN 1). El tráfico de VLAN 100 no llega etiquetado al router. La subinterfaz Gig0/1.100 usa encapsulation dot1Q 100 para identificar paquetes con esa etiqueta; sin etiqueta, no los puede distinguir y el enrutamiento inter-VLAN falla completamente.

    Paso 3b — Verificar show vlan brief tras configurar trunks¶

    Qué se pide: captura del show vlan brief después de configurar los trunks, mostrando que los puertos access siguen correctamente asignados.

    Concepto

    Este apartado verifica que los trunks no han dañado la asignación de puertos access. Si los puertos access (Fa0/1, Fa0/10) siguen apareciendo en sus VLANs correspondientes, la configuración es correcta.

    Error frecuente

    Si en el paso anterior pusiste Fa0/1 o Fa0/10 como trunk, aquí verás que esos puertos ya no aparecen asignados a ninguna VLAN. Debes corregir la configuración de trunks antes de tomar esta captura.

    Comandos de verificación alternativos

    En la recuperación puede pedirse show interfaces trunk en lugar de (o además de) show vlan brief.

    Paso 3c — Ping entre PCs de la misma VLAN¶

    Qué se pide: captura de ipconfig + ping entre dos PCs de la misma VLAN para verificar conectividad L2.

    Concepto

    Este ping prueba que las VLANs están correctamente configuradas y los trunks funcionan. No requiere enrutamiento: ambos PCs están en la misma red lógica.

    Ejemplo: PCD_1 (10.200.0.1) → ping → PCD_2 (10.200.0.2)

    Error frecuente

    Hacer ping a la propia IP del PC (self-ping). Eso siempre responde y no prueba ninguna conectividad. El ping debe ser a otro PC de la misma VLAN.

    Sobre los timeouts en Packet Tracer

    El primer paquete del ping puede salir como timeout por la resolución ARP. Esto es normal. Con que haya al menos una respuesta (Reply), la conectividad está demostrada.

    Reflexión

    Si el ping de este paso falla, ¿qué elementos de configuración deberías revisar? Lista al menos tres posibles causas.

    Respuesta
    1. VLANs no creadas en alguno de los switches del camino.
    2. Puertos access no asignados a la VLAN correcta en los switches de acceso.
    3. Trunks no configurados en los uplinks entre switches.
    4. Los PCs origen y destino están configurados en VLANs distintas por error.
    5. Problema de conectividad física en PT (cable desconectado).

    Paso 4a — Configurar SVIs en el switch L3¶

    Qué se pide: captura mostrando la creación de las SVIs con sus IPs y el comando ip routing en el switch L3.

    Concepto

    Para enrutamiento inter-VLAN en un switch L3 se necesitan dos cosas: 1. ip routing — activa el motor de enrutamiento 2. Una SVI por cada VLAN, con IP y no shutdown

    ! En SWL3_1
ip routing

interface vlan 30
 ip address 192.168.30.254 255.255.255.0
 no shutdown

interface vlan 40
 ip address 192.168.40.254 255.255.255.0
 no shutdown

    Error frecuente

    Configurar las SVIs correctamente pero olvidar ip routing. El switch no enrutará entre VLANs aunque todo lo demás esté bien. En la captura del examen, ip routing debe ser visible en el bloque de comandos o en el show running-config.

    Verificación

    show ip interface brief   ! Vlan30 y Vlan40 deben aparecer up/up con sus IPs
show ip route             ! Deben aparecer las redes C (connected) de las VLANs

    Reflexión

    Una SVI puede estar up/down (up en capa 1, down en capa 2). ¿Qué significa eso? ¿Qué tendrías que hacer para que aparezca up/up?

    Respuesta

    Status up / Protocol down: la interfaz lógica existe y no tiene shutdown (capa 1 ok), pero el protocolo de capa 2 está caído porque no hay ningún puerto físico activo asignado a esa VLAN en el switch. Para que aparezca up/up debe existir al menos un puerto de esa VLAN activo con conexión. En PT: conecta un PC a un puerto access de esa VLAN y asegúrate de que la interfaz del PC está up.

    Paso 4b — show ip interface brief en SWL3_1¶

    Qué se pide: captura del show ip interface brief en el switch L3 mostrando las SVIs activas.

    Salida esperada (ejemplo):

    Interface              IP-Address      OK? Method Status                Protocol
...
Vlan30                 192.168.30.254  YES manual up                    up
Vlan40                 192.168.40.254  YES manual up                    up
GigabitEthernet0/2     10.0.0.1        YES manual up                    up

    Error frecuente

    Las SVIs aparecen up/down (protocolo down). Esto ocurre cuando no hay puertos activos asignados a esas VLANs. Comprueba que los PCs de esas VLANs están conectados y sus interfaces están up.

    Reflexión

    ¿Qué diferencia hay entre el Status y el Protocol en la salida del show ip interface brief? ¿Qué indica cada uno cuando está down?

    Respuesta
    • Status (capa 1): estado físico de la interfaz. administratively down = hay un shutdown manual; down = sin señal o sin conexión.
    • Protocol (capa 2): comunicación bidireccional activa con el otro extremo. down con Status up = conexión física ok, pero sin protocolo de capa 2 funcionando (típico en SVIs sin puertos activos en esa VLAN, o enlaces sin configuración en el extremo remoto).

    Paso 4c — Ping inter-VLAN a través de SWL3_1¶

    Qué se pide: captura de un ping entre dos PCs de VLANs distintas, enrutado por SWL3_1.

    Concepto

    Este ping prueba el enrutamiento inter-VLAN del switch L3. El paquete sube por la SVI de la VLAN origen, el switch lo enruta, y lo entrega por la SVI de la VLAN destino.

    Ejemplo: PCA_2 (192.168.30.2) → ping → PRB_2 (192.168.40.2)

    Error frecuente

    El ping falla porque falta ip routing (ver paso 4a). Si este ping falla pero los del paso 3c funcionaron, el problema está en el enrutamiento, no en las VLANs.

    Reflexión

    ¿Qué ruta sigue el paquete de PCA_2 a PRB_2? ¿Qué dispositivo hace el enrutamiento? ¿Qué IP usa cada PC como gateway?

    Respuesta

    PCA_2 (192.168.30.2) envía el paquete a su gateway 192.168.30.254 (Vlan30 de SWL3_1). SWL3_1 consulta su tabla de rutas: 192.168.40.0/24 es directamente conectada por Vlan40, y lo reenvía por ahí. PRB_2 (192.168.40.2) lo recibe.

    • Dispositivo que enruta: SWL3_1.
    • Gateway de PCA_2: 192.168.30.254
    • Gateway de PRB_2: 192.168.40.254

    Paso 5a — Configurar enlace routed en SWL3_1¶

    Qué se pide: captura mostrando la configuración de la interfaz Gig0/2 del switch L3 como puerto routed (no switchport) con IP.

    Concepto

    Por defecto, todas las interfaces de un switch son puertos de capa 2 (switchport). Para conectar el switch L3 directamente con el router mediante una IP punto a punto, hay que convertir esa interfaz a modo routed con no switchport.

    El otro extremo del enlace también necesita configuración

    El enlace /30 tiene dos extremos. El examen pide captura en SWL3_1, pero R_1 Gig0/0 también debe configurarse para que el enlace quede operativo (normalmente se hace junto con el resto de R_1 en el paso 6a, pero puede hacerse ya):

    ! En R_1
interface GigabitEthernet0/0
 ip address 10.0.0.2 255.255.255.252
 no shutdown

    ! En SWL3_1
interface GigabitEthernet0/2
 no switchport
 ip address 10.0.0.1 255.255.255.252
 no shutdown

    Error frecuente

    Olvidar no switchport. Sin él, la interfaz permanece en modo L2 y no acepta dirección IP. El enlace punto a punto con el router no funcionará.

    Reflexión

    ¿Por qué en este caso no se usa una SVI sino un puerto routed? ¿Cuál es la diferencia entre un puerto routed y una SVI?

    Respuesta

    Se usa un puerto routed porque el enlace con R_1 es punto a punto y no pertenece a ninguna VLAN — es una conexión de red pura entre dos dispositivos L3.

    • SVI: interfaz lógica asociada a una VLAN; actúa como gateway de todos los dispositivos de esa VLAN. No es una interfaz física, no puede usarse para un enlace P2P.
    • Puerto routed (no switchport): convierte una interfaz física en un puerto de capa 3 con su propia IP, sin pertenecer a ninguna VLAN. Ideal para enlaces directos entre routers o entre router y switch L3.

    Paso 5b — show interfaces switchport¶

    Qué se pide: captura del comando show interfaces GigabitEthernet0/2 switchport mostrando que el puerto está en modo routed.

    Concepto

    El comando show interfaces <interfaz> switchport muestra si una interfaz tiene el modo switchport habilitado o no. En un puerto routed (después de no switchport), la salida indica Switchport: Disabled.

    show interfaces GigabitEthernet0/2 switchport

    Salida esperada: 

    Name: Gig0/2
Switchport: Disabled
...

    Error frecuente

    Usar show interfaces GigabitEthernet0/2 (sin la palabra clave switchport). Este comando muestra estadísticas de la interfaz (paquetes, errores, etc.) pero no muestra el campo "Switchport: Disabled". La captura no será válida para este apartado.

    Reflexión

    ¿Qué diferencia hay en la salida entre show interfaces Gig0/2 y show interfaces Gig0/2 switchport? ¿Por qué es importante usar el comando correcto?

    Respuesta
    • show interfaces Gig0/2: muestra estadísticas físicas de la interfaz (estado, velocidad, duplex, contadores de paquetes y errores). No indica si switchport está habilitado o no.
    • show interfaces Gig0/2 switchport: muestra la configuración de capa 2: si switchport está habilitado (Enabled) o deshabilitado (Disabled), el modo (access/trunk) y la VLAN asignada. En un puerto routed, aparece Switchport: Disabled.

    Es importante usar el correcto porque el apartado 5b pide demostrar explícitamente que el puerto está en modo routed, y solo el segundo comando lo hace visible.

    Paso 6a — Configurar router-on-a-stick en R_1¶

    Qué se pide: captura de la configuración de las subinterfaces del router con encapsulation dot1Q e IPs.

    Concepto

    En el router-on-a-stick, cada subinterfaz se asocia a una VLAN mediante encapsulation dot1Q <id_vlan>. Este comando debe ejecutarse antes de asignar la IP. La interfaz física no necesita IP pero sí debe estar en no shutdown.

    ! En R_1
interface GigabitEthernet0/1
 no shutdown          ! La física debe estar activa

interface GigabitEthernet0/1.100
 encapsulation dot1Q 100
 ip address 10.100.255.254 255.255.0.0

interface GigabitEthernet0/1.200
 encapsulation dot1Q 200
 ip address 10.200.255.254 255.255.0.0

    Error frecuente

    Omitir encapsulation dot1Q. Sin este comando, la subinterfaz no sabe qué tráfico VLAN debe procesar. El router no enrutará correctamente el tráfico de esas VLANs.

    Error frecuente

    IP de subinterfaz incorrecta. Para una red /16 como 10.100.0.0/16, la última IP host es 10.100.255.254, no 10.100.0.254.

    Verificación

    show ip interface brief
! Gig0/1.100 y Gig0/1.200 deben aparecer up/up con sus IPs

    Reflexión

    ¿Qué sucede si pones encapsulation dot1Q 100 en la subinterfaz que debería ser VLAN 200, y viceversa? ¿Cómo afectaría a la conectividad?

    Respuesta

    El tráfico de VLAN 100 llegaría a la subinterfaz con la IP de VLAN 200, y el de VLAN 200 a la de VLAN 100. Los PCs no encontrarían su gateway en la subinterfaz correcta. Los pings entre VLANs fallarían porque el router respondería desde la IP equivocada. En la práctica, los PCs ni siquiera podrían alcanzar su gateway predeterminado.

    Paso 6b — show ip interface brief en R_1¶

    Qué se pide: captura del show ip interface brief en R_1 mostrando las subinterfaces activas.

    Salida esperada (ejemplo):

    Interface              IP-Address       OK? Method Status                Protocol
GigabitEthernet0/0     10.0.0.2         YES manual up                    up
GigabitEthernet0/1     unassigned       YES unset  up                    up
GigabitEthernet0/1.100 10.100.255.254   YES manual up                    up
GigabitEthernet0/1.200 10.200.255.254   YES manual up                    up

    Error frecuente

    Las subinterfaces aparecen como administratively down. Esto ocurre cuando la interfaz física (Gig0/1) tiene shutdown. Asegúrate de ejecutar no shutdown en Gig0/1.

    Reflexión

    Si Gig0/1 está up pero Gig0/1.100 está down, ¿cuál puede ser la causa? ¿Y si ambas están administratively down?

    Respuesta
    • Gig0/1 up pero Gig0/1.100 down: la subinterfaz tiene shutdown manual, o el trunk del switch no está activo (la VLAN no llega al router).
    • Ambas administratively down: la interfaz física Gig0/1 tiene shutdown. Ejecuta no shutdown en interface GigabitEthernet0/1. Al activarla, las subinterfaces deberían subir automáticamente.

    Paso 6c — Ping inter-VLAN a través de R_1¶

    Qué se pide: captura de un ping entre PCs de las VLANs gestionadas por el router (VLANs 100 y 200).

    Ejemplo: PCC_2 (10.100.0.2) → ping → PCD_2 (10.200.0.2)

    Concepto

    Este ping prueba que el router-on-a-stick funciona correctamente. El paquete sale por VLAN 100, llega al router por la subinterfaz Gig0/1.100, el router lo enruta, y lo envía de vuelta por Gig0/1.200 hacia VLAN 200.

    Reflexión

    Para que este ping funcione, ¿qué configuraciones previas deben estar correctas? Enumera al menos 4.

    Respuesta
    1. VLANs 100 y 200 creadas en SW_2 y SW_3 con puertos access asignados correctamente.
    2. Trunk activo en Gig0/1 de SW_2 (uplink hacia R_1).
    3. Subinterfaces del router con encapsulation dot1Q correcto e IPs correctas (x.x.255.254/16).
    4. Interfaz física Gig0/1 del router en no shutdown.
    5. IPs de los PCs configuradas con el gateway correcto (x.x.255.254).

    Paso 7a — Rutas estáticas en R_1¶

    Qué se pide: captura mostrando los comandos ip route en R_1 y el show ip route con las rutas activas.

    Concepto

    R_1 necesita rutas estáticas para llegar a las redes de las VLANs 30 y 40, que están detrás de SWL3_1. El next-hop es la IP de SWL3_1 en el enlace punto a punto (la IP del lado de SWL3_1 en el /30).

    Rutas estáticas también en SWL3_1

    El enrutamiento es bidireccional: si R_1 no sabe llegar a las redes de SWL3_1, y SWL3_1 no sabe llegar a las redes de R_1, los pings del bloque 8 fallarán en ambos sentidos. Configura también las rutas en SWL3_1 (se verificarán en el paso 7b):

    ! En SWL3_1
ip route 10.100.0.0 255.255.0.0 10.0.0.2
ip route 10.200.0.0 255.255.0.0 10.0.0.2

    ! En R_1
ip route 192.168.30.0 255.255.255.0 10.0.0.1
ip route 192.168.40.0 255.255.255.0 10.0.0.1

    Verificación en show ip route: 

    S    192.168.30.0/24 [1/0] via 10.0.0.1
S    192.168.40.0/24 [1/0] via 10.0.0.1

    Error frecuente

    Los comandos ip route se teclean correctamente pero las rutas no aparecen como S activas en el show ip route. Causa habitual: el next-hop (10.0.0.1) no es alcanzable porque el enlace punto a punto no está configurado correctamente (paso 5a o la IP del lado R_1 en Gig0/0 es incorrecta).

    Verificación

    La captura del show ip route es la prueba definitiva. Si no ves la S junto a las redes destino, las rutas no están activas aunque hayas tecleado los comandos.

    Reflexión

    ¿Qué rutas directamente conectadas (C) debería ver R_1 en su tabla de enrutamiento? Enuméralas antes de mirar la solución.

    Respuesta

    R_1 tiene tres redes directamente conectadas:

    • C 10.0.0.0/30 — por Gig0/0 (enlace P2P con SWL3_1)
    • C 10.100.0.0/16 — por Gig0/1.100 (subinterfaz VLAN 100)
    • C 10.200.0.0/16 — por Gig0/1.200 (subinterfaz VLAN 200)

    Las redes 192.168.30.0/24 y 192.168.40.0/24 no son directamente conectadas a R_1 (están detrás de SWL3_1), por eso necesitan rutas estáticas S.

    Paso 7b — show ip route en SWL3_1¶

    Qué se pide: captura del show ip route en SWL3_1 mostrando la tabla de enrutamiento completa.

    Tabla esperada (ejemplo):

    C    10.0.0.0/30         via GigabitEthernet0/2
S    10.100.0.0/16       [1/0] via 10.0.0.2
S    10.200.0.0/16       [1/0] via 10.0.0.2
C    192.168.30.0/24     via Vlan30
C    192.168.40.0/24     via Vlan40

    Concepto

    • C (Connected): redes directamente conectadas al dispositivo
    • S (Static): rutas configuradas manualmente con ip route
    • El next-hop de las rutas S hacia las VLANs del router es la IP de R_1 en el enlace punto a punto

    Reflexión

    ¿Por qué SWL3_1 necesita rutas estáticas hacia 10.100.0.0/16 y 10.200.0.0/16? ¿No podría llegar a esas redes directamente?

    Respuesta

    SWL3_1 no tiene ninguna interfaz en las redes 10.100.0.0/16 ni 10.200.0.0/16. Esas redes están al otro lado de R_1. Sin rutas estáticas, SWL3_1 no sabría por dónde enviar los paquetes destinados a esas redes. Las rutas estáticas le indican: «para llegar a 10.100.0.0/16 o 10.200.0.0/16, envía el paquete a R_1 (next-hop 10.0.0.2)».

    Paso 8 — Pruebas de conectividad entre redes remotas¶

    Qué se pide: capturas de pings entre dispositivos de redes distintas, verificando que el enrutamiento end-to-end funciona.

    Concepto

    Estos pings prueban que toda la topología funciona de extremo a extremo: VLANs, trunks, SVIs, enlace routed, router-on-a-stick y rutas estáticas. Si algún paso anterior falla, estos pings lo evidenciarán.

    Ejemplo de pares probados en la prueba original:

    Apt. Origen Destino Ruta
    8a PCA_1 (192.168.30.1) PCC_1 (10.100.0.1) SWL3_1 → R_1
    8b PCA_2 (192.168.30.2) PCD_1 (10.200.0.1) SWL3_1 → R_1
    8c PCB_1 (192.168.40.1) PCC_2 (10.100.0.2) SWL3_1 → R_1
    8d PCD_2 (10.200.0.2) PRB_2 (192.168.40.2) R_1 → SWL3_1
    8e PCA_1 (192.168.30.1) R_1 Gig0/0 (10.0.0.2) SWL3_1
    8f PCD_2 (10.200.0.2) SWL3_1 Gig0/2 (10.0.0.1) R_1

    Sobre los timeouts

    En Packet Tracer el primer paquete del ping puede salir como timeout (.) por la resolución ARP. No es un error. Con una sola respuesta (!) la conectividad está demostrada.

    Error frecuente

    Hacer ping al destino incorrecto. Por ejemplo, hacer ping a 10.200.0.2 cuando se pide a 10.100.0.2. Lee con cuidado el enunciado de cada apartado para identificar el par correcto.

    Error frecuente

    No incluir captura de algún ping. Cada apartado (8a, 8b, 8c, 8d, 8e, 8f) necesita su propia captura o estar visible en la misma captura junto a los demás pings del mismo PC.

    Reflexión

    Si el ping del apartado 8a falla pero los de los pasos 4c y 6c funcionaron, ¿qué parte de la configuración falta? ¿En qué dispositivo está el problema?

    Respuesta

    Si 4c (inter-VLAN por SWL3_1) y 6c (inter-VLAN por R_1) funcionan pero 8a (PCA_1 → PCC_1, entre redes remotas) falla, la conectividad local es correcta en ambos bloques, pero falta el enrutamiento entre ellos. El problema está en las rutas estáticas (Paso 7): R_1 no sabe llegar a 192.168.30.0/24 (falta la ruta S en R_1) y/o SWL3_1 no sabe llegar a 10.100.0.0/16 (falta la ruta S en SWL3_1).

    5. Checklist final de autoevaluación¶

    Antes de entregar, repasa esta lista. Si algún punto no puedes marcarlo, vuelve al apartado correspondiente.

    Direccionamiento¶

    • [ ] La tabla de redes tiene todas las máscaras correctas (0a)
    • [ ] Las IPs de subinterfaces y SVIs son la última IP host del prefijo, no la primera
    • [ ] La puerta de enlace de cada PC coincide con la SVI/subinterfaz de su VLAN

    VLANs¶

    • [ ] Solo he creado las VLANs necesarias en cada switch (las que tiene en su segmento)
    • [ ] He usado la CLI, no la GUI, para crear las VLANs
    • [ ] Los puertos de acceso están asignados a las VLANs correctas
    • [ ] El show vlan brief (o show interfaces trunk) confirma la configuración

    Trunk¶

    • [ ] Los trunks están configurados en los uplinks (hacia el router y hacia otros switches)
    • [ ] Ningún puerto de acceso (hacia PCs) está en modo trunk
    • [ ] El uplink hacia el router (RoaS) está en modo trunk

    Switch L3 — SVI e ip routing¶

    • [ ] He ejecutado ip routing en el switch L3
    • [ ] Las SVIs tienen IP y están en no shutdown
    • [ ] El show ip interface brief muestra las SVIs en estado up/up
    • [ ] El show ip route muestra las redes de las VLANs como C (connected)

    Enlace routed¶

    • [ ] He ejecutado no switchport en la interfaz punto a punto del L3
    • [ ] La interfaz tiene la IP correcta del bloque /30
    • [ ] El show interfaces <int> switchport muestra "Switchport: Disabled"

    Router-on-a-stick¶

    • [ ] La interfaz física del router está en no shutdown
    • [ ] Cada subinterfaz tiene encapsulation dot1Q <id_vlan> antes de la IP
    • [ ] Las IPs de las subinterfaces son correctas (última IP del prefijo)
    • [ ] El show ip interface brief muestra las subinterfaces up/up

    Rutas estáticas¶

    • [ ] He configurado rutas estáticas en ambos sentidos (R_1 hacia las redes del L3, y SWL3_1 hacia las redes del router)
    • [ ] El show ip route confirma que las rutas S están activas (no solo tecleadas)

    Capturas¶

    • [ ] Tengo captura de todos los apartados, incluidos los pings 8a–8f
    • [ ] Los pings muestran al menos una respuesta (!) aunque haya timeouts (.)
    • [ ] Los destinos de los pings son los correctos según el enunciado

    Recuerda: en la recuperación los valores concretos (IPs, VLANs, nombres de interfaz, comandos de verificación) pueden cambiar. Este checklist y los conceptos de la guía aplican a cualquier variante de la prueba.

    SWL3_1 — Switch de capa 3¶

    enable
configure terminal

! Activar enrutamiento (OBLIGATORIO antes de cualquier SVI)
ip routing

! Crear VLANs
vlan 30
 name VLAN_30
vlan 40
 name VLAN_40
exit

! Trunk hacia SW_1
interface GigabitEthernet0/1
 switchport mode trunk

! Enlace punto a punto hacia R_1 (puerto routed, sin switchport)
interface GigabitEthernet0/2
 no switchport
 ip address 10.0.0.1 255.255.255.252
 no shutdown

! SVIs: una por VLAN, actúan como gateway de sus PCs
interface vlan 30
 ip address 192.168.30.254 255.255.255.0
 no shutdown
interface vlan 40
 ip address 192.168.40.254 255.255.255.0
 no shutdown

! Rutas estáticas hacia las redes que gestiona R_1

ip route 10.200.0.0 255.255.0.0 10.0.0.2

end
